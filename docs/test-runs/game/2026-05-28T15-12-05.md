## Findings

- **NEW REGRESSION — Step 04b (deep-link name drop)**: When a user lands on
  the join screen via `?join=CODE`, types their name into `#join-name-input`,
  and submits, the server receives the random placeholder name instead of
  the typed one. Root cause: `joinGame()` in `static/js/landing.js` calls
  `showLoading('Loading…')` (which swaps the active screen to `#loading`)
  *before* `getName()` runs inside `connectWS()`'s WS-open callback. By then
  the active screen is `loading`, not `join`, so `getName()` falls through
  to reading `#name-input` (the landing input, which is empty on a
  deep-link load) and returns `state.randomNamePlaceholder`. Fix: capture
  the name *before* calling `showLoading`. Same issue likely applies to
  `createGame()` but harder to hit because the landing input is the one
  that's populated. To repro: load `/?join=XXXXX`, type a name, submit —
  server logs show the placeholder.

- **NEW REGRESSION — Step 05 (invalid code stuck-loading)**: After a join
  fails with `error: "Game not found"`, the user is left on the Loading…
  screen with no visible error. Root cause: the error handler in
  `static/js/ws.js` `case 'error'` checks `document.getElementById('join')
  .classList.contains('active')` — but by the time the error arrives, the
  active screen is `loading` (swapped by `showLoading` before the WS round
  trip). Falls through to `setError(msg.msg)` which writes to
  `#landing-error`, but `landing` isn't visible either. Net: user sees
  "Loading…" forever. Fix: either swap back to `landing`/`join` on error,
  or check both `landing` and `loading` active when routing the error.

- **NEW REGRESSION — Step 07 (late-join stuck-loading)**: Same root cause
  as #05. Server's "Game already in progress" reaches the client but
  displays on the inactive `#landing-error`, user stays on Loading…
  indefinitely.

- **STATE QUIRK — Step 17 (disconnect)**: `state.currentState` doesn't get
  updated when the loading-screen branch of `showFor()` fires. The visible
  UX is correct (loading screen with right name), but `_state.currentState`
  reflects the *previous* game-screen render. Not user-visible but worth
  noting — if any future feature reads `state.currentState` for "is X
  disconnected", it'll be wrong. Fix: assign `state.currentState = msg` at
  the top of `showFor()` so it always tracks the latest server state.

- **EXPECTED — Step 18 (reconnect)**: Same 30s-grace-window artifact as
  the prior 2 runs. The direct-WS probe confirms the server-side reconnect
  logic still works (`type:state, disconnected:False`). The Playwright UI
  reconnect is borderline because each tool call adds ~500 ms and Beta was
  closed early.

- **EXPECTED — Step 12 (broadcast timing)**: 877 ms is below the skill's
  1000 ms lower bound, joining the prior 950 ms. The skill's lower bound
  should probably move to 800 ms (with a sub-condition for new-matches
  rolls). New series: 1029, 1491, 1589, 1668, 1635, 950, **877** ms.

  screen *before* the WS roundtrip completes breaks any code in the
  WS-open callback that reads "what screen is the user on?" — that's
  `getName()` (read), the error handler (write), and probably anything
  similar elsewhere. A single fix to either:
    (a) capture screen-dependent values before showLoading, or
    (b) route error/state messages back to the originating screen
  would clear all three.
