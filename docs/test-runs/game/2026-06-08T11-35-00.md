## Findings

- **Step 22 FAIL — winner overlay closed at 1153ms in Round 8 (Alpha/host).**
  - **All other entries were 2987–3042ms** (12 of 13 overlays). One anomalous entry: Alpha Winner R8 = 1153ms.
  - **Root cause:** `showFor()` in `static/js/net.js` calls `hideWinner()` unconditionally (lines 213, 221, 234, 242). When a `state` broadcast from Beta's roll arrives after Alpha's reveal has completed (but while the winner overlay is still open), `handleMessage` routes it to `showFor()` → `hideWinner()`, closing the overlay early.
  - **Why now?** The `5916a72` commit (2026-06-08, "Show the winner overlay immediately after scatter reveal") moved the overlay open point ~350ms earlier (removed the 350ms post-landing dwell). This widened the "post-reveal, overlay-open, awaitingAck=false" window by 350ms, increasing the probability that an opponent's roll broadcast lands in that window.
  - **The existing fix** (drop `postRevealState` in `tryReveal` when `pendingWinName` is set) only guards the mid-reveal window, not the post-reveal window.
  - **Proposed fix:** In `handleMessage` (`static/js/net.js`), when a `state` frame arrives and `document.getElementById('winner-overlay').open === true`, stash it (e.g. in `state.postRevealState`) instead of calling `showFor()`. Or in `showFor()`, skip `hideWinner()` if the overlay is open and the new state isn't a round-advance (same `round_num`).
  - **Not the < 1000ms flash regression** — 1153ms is above that threshold. A distinct but related bug.
  - Per-round join consistency was perfect: exactly one Winner + one Loser every round (R3–R8), no double-wins.

## Notes

- **`_state` access requires versioned dynamic import.** The app loads modules with cache-busting: `state.js?v=5071dad0`. Importing without the version hash creates a fresh module instance (all nulls). The correct pattern for all steps that use module state:
  ```js
  const stateUrl = performance.getEntriesByType('resource').find(r => r.name.includes('state.js'))?.name;
  const url = new URL(stateUrl);
  const m = await import(url.pathname + url.search);
  const s = m.state;  // use s instead of _state
  ```
  This works in both dev (separate files) and prod (bundle exposes state at the same URL).
  **NOTE: prod bundles state.js into app.js — this pattern only works for dev. In prod, state.js is not a separate resource.**

- **Dev stack required for WebSocket testing.** Prod `.env.prod` has `ALLOWED_ORIGINS=https://dev.tensies.app`, which blocks WS connections from localhost. Always use `docker compose up -d` (dev) for local Playwright tests.

- **Postgres volume conflict.** The pg_data volume from the prod stack (password `localtest`) conflicted with dev stack credentials (password `tensies`). Resolved by dropping and recreating the volume (`docker volume rm tensies_pg_data`). This is a one-time issue from mixing dev/prod stacks.

- **Step 11 broadcast timing = 1677ms** — within the new 800–2000ms window. Higher than recent runs (934ms in June 7). Fewer locked dice at roll time means longer reveals; more locked dice → shorter broadcast delay.

- **Step 13 sticky overlay** — 3029ms, 30 spacebar fires, closed cleanly. Holds.

- **Beta session re-seeding required at Steps 16 and 19** (as anticipated — the throwaway tabs from Steps 5/7 clobber instance #2's localStorage).
