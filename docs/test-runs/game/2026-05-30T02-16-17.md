## Notes

- **Self-update**: the skill was stale (4 pause commits postdated it: `7a16a6d`,
  `57d3a47`, `9bfbae5`, `242a5d7`). Added three pause regression steps (15 host toggle
  + host-only + non-host wait screen; 16 status panel / countdown / drops-suspended /
  host-returns-to-open-menu; 17 resume) and renumbered the disconnect/reconnect/host/
  animation/console steps to 18–22. Summary table and counts updated to 22.
- **Pause feature verified end-to-end**: host-only toggle (guest's `#menu-pause-btn`
  is `hidden`), non-host loading text "Waiting for Alpha to resume the game", live
  countdown ticked 60:00 → 59:58, "X of Y connected" went 2/2 → 1/2 → 2/2 as the guest
  left and returned, a host returning from reconnect landed on the board with the menu
  auto-opened, and resume closed the menu + re-enabled rolling.
- **Drops-suspended check (Step 16)**: the client's *correct* auto-reconnect made the
  transient "1 of 2"/disconnected window unobservable on the first pass (the guest
  reconnected before the assert read). Made it deterministic by setting
  `_state.reconnecting = true` before `ws.close()` — that models a player who hasn't
  returned yet (suppresses auto-reconnect + the connection-lost alert) — then reload to
  reconnect. Same `goAway()` helper used for Steps 18 and 20.
- **Rate limit (Step 10)**: rewrote to a deterministic throwaway *solo* game (fresh
  round, `matched=0`, so the first raw frame can't win and the second is guaranteed
  `< MIN_ROLL_INTERVAL` later → "Slow down"). The original main-game version was flaky
  because Alpha's round state (round_over / high matched) varied by the time it ran.
  Server logged `roll … RATE LIMIT`; client state stayed clean.
- **Broadcast timing**: 1639 ms from roll click to Beta detecting the state change —
  within the 1000–2000 ms animation window (prior runs: 1491, 1589 ms).
- **Winner overlay**: captured via MutationObserver — winner "Alpha", sub "Next up:
  roll for 5s". Sticky-overlay spacebar guard held (openWindow ≈ 3000 ms, 30 space
  fires, overlay closed).
- **Screenshot `16-pause-status.png` shows the menu over a "Reconnecting…" backdrop** —
  this is a View Transition cross-fade snapshot caught mid-fade while the host's
  reconnect settled (`.screen:not(.active){display:none}` guarantees only one screen is
  logically active). Benign capture artifact; `15-paused.png`/`17-resumed.png` render
  cleanly and all Step 16 assertions passed.
- **Console**: only `favicon.ico` 404 (benign) across both contexts. No JS errors, no
  unexpected WS closes.

## Watch next run
- **Pause render races**: `renderMenu`/`syncPaused` run on a rAF after `currentState`
  updates, so assert on rendered DOM (label text, `#menu-pause-status` visibility), not
  the `paused` flag — reading the flag is one frame too early (bit Steps 15/16 once).
- **`goAway()` pattern**: any "player is offline" assertion must suppress auto-reconnect
  first, or the client reconnects before you can observe the disconnected state.
- **Telemetry untested**: Postgres-backed rollups, `/metrics`, Grafana Live, and
  `pause_timeout` game-end cleanup were not exercised here. The pause cap end-of-game
  path (fatal `error` + cleanup) is covered by a server unit test but not by this suite.
- **Pause cap**: the 1 h `PAUSE_MAX` countdown→game-end wasn't run to completion (would
  take an hour); only the countdown display and drop-suspension were checked.
