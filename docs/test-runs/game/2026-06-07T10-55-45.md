## Findings

- **Skill self-updated mid-run (3 stale assertions corrected against current code):**
  1. **Step 12** referenced `#winner-sub` ("mentions the next target number"). That
     element no longer exists — the overlay is now a banner `Round <#winner-round>
     <#winner-banner-suffix>` + `#winner-name`. Updated the capture + assertions to
     read `#winner-banner-suffix` (Winner/Loser) and `#winner-round`.
  2. **Step 14** claimed the target cycle is **6→5→4→3→2→1** ("one step down"). Source
     of truth `next_target()` in `server/game.py` is `t % 6 + 1` = **1→2→3→4→5→6→1**
     (one step up). Observed live: R1 t1 → R2 t2 → R3 t3. Fixed the assertion.
  3. **Step 18** prescribed a bare `_state.ws.close()` to drop a player, but on a live
     page that immediately fires `handleWsClose → maybeReconnect()` (creds still in
     localStorage), so the peer never reaches the loading screen. Updated to null
     `_state.ws.onclose` before closing — a clean drop that leaves creds intact for the
     Step 19 reload. Same technique applied to the Step 20 host drop.
  4. **Step 16** Beta reload note: added a re-seed fallback (the Step 5/7 throwaway tabs
     share instance #2's persistent profile and clear Beta's session keys; the Step 16
     reload hits this before Step 19 does).

## Notes

- **Step 11 broadcast timing = 934ms** — under the skill's 1000ms nominal lower bound
  but far above the 200ms regression threshold. This continues the documented trend
  (2026-05-30 saw 877ms, 988ms): when many dice are already locked at roll time, the
  reveal animation is shorter, so the peer sees the broadcast sooner. Here the roll was
  at 6/10 matched. Not a regression. The 2026-05-30 log's standing suggestion to lower
  the bound to ~800ms still stands.
- **Step 22 (flash regression) — the headline check, clean.** Both instances ran
  staggered auto-rollers (350ms host / 270ms guest) across rounds 3→9. Every one of the
  14 recorded overlays (7 per client) stayed open **2999–3028ms**; **zero** entries
  < 1000ms, **zero** Winner flashes. Per-round join is consistent — exactly one Winner +
  one Loser every round, complementary across the two clients (R3 A-win/B-lose,
  R4 A-lose/B-win, …). The 2026-06-07 `animations.js` fix (drop the stashed
  `postRevealState` when a win is shown, rather than routing it through `showFor()`)
  holds under sustained fast-opponent play.
- **Step 13** spacebar-hammer: 27 rolls, 30 spacebar fires, overlay window 3006ms,
  closed cleanly — sticky-overlay regression stays fixed.
- Win tally at end of run reflected the auto-roller races; no double-wins observed
  (single Winner per round in the Step 22 logs is the live evidence of the Lua
  win compare-and-set holding).
