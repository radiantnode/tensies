## Findings

- **FAIL 06 — Invalid code rejection** (newly introduced / existing bug): "Game not found" error written to `#landing-error` instead of `#join-error`. Since the join screen is active when the server responds, `#landing-error` is hidden and the user sees nothing. Fixed immediately after this run in `static/game.js` — the `error` message handler now checks which screen is active and routes to `#join-error` when appropriate.

- **FAIL 08 — Join-after-start rejection** (same root cause as #06): "Game already in progress" also routed to `#landing-error`. Fixed by the same change.

## Notes

- `window.currentState`, `window.myId`, etc. do not work in `browser_evaluate` snippets — these are `let` variables scoped to the module, not properties of `window`. Must use bare `currentState`, `myId`, `rolling`, etc. The skill's evaluate examples were updated to reflect this.

- Broadcast timing measured at **1589ms** from roll click to Tab 2 seeing the state change. The expected animation window is 1320–1720ms (200ms gather + 300–700ms shake + 820ms reveal/pop). 1589ms lands in the middle of that range — correct behaviour.

- The tab-switch approach for the <200ms broadcast timing check is unreliable: Playwright tool call overhead alone can be 500ms+. Used a setInterval polling watcher installed on Tab 2 before the roll click, with timestamps recorded inside the page, to get an accurate measurement.

- Winner overlay auto-advances in ~4 seconds. When rolling to win inside an async `evaluate` loop (30s duration), the overlay appears and dismisses before the loop returns. Used a `MutationObserver` on `#winner-overlay` to confirm it fired with the correct content. Screenshot capture of the live overlay requires breaking out of the roll loop when matched===10 and pausing briefly before the evaluate returns.

- Rate limiting appears to be enforced client-side (button disables on first click, blocking the second click from sending a WS message). No rate limit warning appeared in browser console or server logs for the double-click test. The double-click resulted in only one roll being processed, which is the correct outcome.

- Only console error across the entire run was a `404` for `/favicon.ico` — benign and consistent across all tabs.
