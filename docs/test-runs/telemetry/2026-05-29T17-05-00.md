## Notes

- **Auto-reconnect fires when sockets are closed in Step 5.** Calling `_state.ws.close()` on both clients triggers the reconnect loop — both players reconnected successfully before the 60-second grace period expired. This is correct behaviour but means `game_ended` never fires, so `games.status` stays 'playing' and `games.round_count = 0` at the time of the Step 7 check. Not a bug. Step 7 criteria updated to reflect this.
- **connection_opened / connection_closed / session_started / session_ended events are stored without `game_code`** — they appear in the Postgres baseline counts but are not returned by a `WHERE game_code = 'JIUUF'` filter. This is by design (they fire before a player joins a game). Confirmed via `user_id` filter: session_started=4, session_ended=2 for our test players.
- **`reconnected: 2` events for game JIUUF** — expected, caused by the auto-reconnect after socket close.
- **Connections dashboard "error" false positive** — the "Disconnect reasons (1h)" pie chart has a slice labeled `client error` (the metric label value). The panel is displaying correctly; the error-detection regex matched the slice label text.
- **Skill self-corrected during run:** two SQL bugs found and fixed: `v_dice_roll_distribution.count` → `.rolls` (the column is named `rolls`), and `ORDER BY check` → `ORDER BY label` (`check` is a reserved word in Postgres).
