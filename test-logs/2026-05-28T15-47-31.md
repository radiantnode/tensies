## Findings

- **NEW FAIL — Step 19 (client reconnect doesn't send action)**:
  Beta's WS was closed at 20:38:41, my tight reconnect call ran at
  20:38:54 (well within the 30 s grace), but server logs show only:
    ```
    20:38:41  disconnect  pid=8c87834e  player=Beta  game=BXZRC
    20:38:41  connect  pid=31a2594d  session=8e0d3384
    20:38:41  disconnect  pid=31a2594d  player=?  game=none
    ```
  The auto-reconnect WS (from `handleWsClose`) connects but disconnects
  *without ever sending the `reconnect` action*. The explicit
  maybeReconnect at 20:38:55 also opened a WS that disconnected with
  `player=?`. Final state on the page: `state.myId =
  "d0030ebc-eb24-…"` — a pid that doesn't match any server session in
  this run. `localStorage.tensies_pid` was `null` but
  `localStorage.tensies_code` still held `"BXZRC"` — inconsistent
  (clearSession is supposed to remove both atomically).
  
  Where to look:
  - `static/js/ws.js#attemptReconnect` — `onopen` is supposed to send
    `{action:'reconnect',player_id:pid,game_code:code}`. Either it never
    fires, or fires but the send fails. Server-side processing is
    confirmed working via direct-WS probe.
  - The mystery `d0030ebc-…` pid suggests `handleMessage('welcome')`
    ran on the reconnect WS at some point, reassigning `state.myId` and
    `localStorage.tensies_pid` to the *new* session's pid. Then the
    second reconnect attempt sent that wrong pid → server rejected →
    expireSession → clearSession (which should remove both, but only
    pid was removed).
  - Recommended debug path: add a console.log inside `onopen`,
    `onmessage` (with msg.type), and right before `state.ws.send`. Run a
    single disconnect→reconnect cycle and trace the order of events.
  
  Direct-WS probe at 20:39:23 (TestX → close → reconnect) **passed
  cleanly**: `type:state, disconnected:False`. Server reconnect is
  unaffected — this is purely a client-side regression.
