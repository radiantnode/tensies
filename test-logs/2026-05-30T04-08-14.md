## Notes

- **#12's earlier "batch flakiness" was actually a test-script bug, not a product issue or
  flake.** The test dropped a guest (g2) to verify the "2 of 3 connected" count, then
  resumed and asserted the other guest (g1) was on the game board. But with g2 still
  disconnected, g1 *correctly* shows the normal "Waiting for Cy to reconnect…" loading
  screen — so `active(g1,'game')` rightly timed out. The labeled `active(game); on=loading`
  error surfaced this. Earlier standalone runs passed only because they stopped before the
  resume step. Fixed the test to reconnect g2 (back to "3 of 3") before resuming; now g1
  and g2 both return to the board. Product behavior was correct throughout.
- **#15** verified via the deterministic client path (point the guest's saved code at a
  missing game, drop the socket → "Game not found" → expireSession → landing), avoiding the
  flaky in-sandbox server restart. Identical to the post-restart condition.
