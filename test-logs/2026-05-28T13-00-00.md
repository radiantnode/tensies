## Findings

- **Step 13 (broadcast timing) — 950 ms, below the skill's 1000-2000 ms
  band but well above the 200 ms regression threshold.** The roll
  before this measurement had already filled all 6-matches Alpha was
  going to get this round, so the test roll added zero new matches.
  When `newlyMatchedCount === 0`, `updateDiceInPlace` skips the 500 ms
  `popT` block and fires `onComplete` (which sends `roll_done`)
  immediately after the reveal phase. Computed lower-bound for that
  path: shake(500-900) + scatter(320) + reveal(300) ≈ 1120-1520 ms, and
  the observed 950 ms is below even that — likely network/timing noise
  on a fast shake. Not a regression of the delayed-broadcast invariant
  (which the 200 ms threshold guards). Consider lowering the skill's
  expected lower bound to ~800 ms or adding a sub-condition for
  no-new-matches rolls.

  Prior measurements (for reference): 1029, 1491, 1589, 1668, 1635 ms.
  New series including this run: **add 950 ms**.

- **Step 18 (reconnect UX) — same 30s-grace-window artifact as the
  prior run.** Sequence: Beta closed at 13:00:11, my reconnect call
  landed at 13:00:42 (31 s later) because tab-switch + sleep + evaluate
  ate the budget. Server's drop_player fired at 13:00:41 and removed
  Beta exactly one second before reconnect arrived; reconnect was
  correctly rejected with "Your session expired".

  Verified the underlying reconnect logic via a direct two-step
  websocket (create → close → reconnect within grace): passed cleanly,
  state echo shows `disconnected=False`. So the code is fine; this is
  test-tooling latency. Same diagnosis and same workaround as the
  prior log.

- **Skill updated mid-run (Step 0):** the modernize commit replaced the
  `#create-btn` / `#join-btn` ids with `type="submit"` inside forms.
  Skill steps 3, 4, 5, 7 updated to either submit the form via
  `requestSubmit()` or click `#<form-id> button[type="submit"]`.

  state. The skill's player-card checks now look at `getAttribute('name')`
  / `hasAttribute('is-me')` instead of `.player-mini-name.textContent`.

  "you" badge `<span class="player-mini-you">` is `hidden` for cards
  that aren't me, but its text still shows up in `textContent`. Used
  `.hidden` + `.hasAttribute('is-me')` checks instead.

  not class `.visible` (the modernize commit moved to `showModal()` /
  `.close()`). Updated step 13 (round transition check), step 15
  (disconnect overlay), and step 12 (winner MutationObserver) — the
  observer now watches all attribute changes since `open` is reflected
  as an attribute on dialog elements.
